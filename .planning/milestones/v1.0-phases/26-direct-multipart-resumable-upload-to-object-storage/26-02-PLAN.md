<task type="auto">
  <name>Task 1: Implement a browser upload runner with cancel + retry + progress callbacks</name>
  <files>src/lib/uploads/runPdfUploadSession.ts</files>
  <action>
Create `runPdfUploadSession` as a small orchestration helper that takes:
- `file: File`
- `suggestedSuffix: string` (do NOT pass raw file name; pass a sanitized coarse suffix like `"document"` or `"import"`)
- callbacks:
  - `onProgress({ uploadedBytes, totalBytes, percent })`
  - `onStatus({ state })` for transitions (idle → init → uploading → finalizing → done/failed/cancelled)
- returns an object with:
  - `promise` resolving to:
    - `{ ok: true, upload: { uploadId, key, expiresAt } }`, OR
    - `{ ok: false, uploadCapability: { configured: boolean, providerReady: boolean, mode: "local-only" | "direct-upload", reason?: "not_configured" | "provider_not_ready" }, message?: string }`
  - `cancel()` (per D-15)

Implementation constraints:
- Same-session resume only (D-07). Keep state in memory; do not persist resume info to storage.
- Auto retry/resume within the same session (D-13):
  - On transient fetch/PUT failures, retry with exponential backoff and jitter using the existing retry helper patterns (search for `withRetries` usage; align with `pipelineStageRetry` semantics).
  - Resume means “continue remaining parts/chunks” (not restart from 0) when possible; if not possible, restart upload and reflect that in progress.
- Multipart vs resumable-session abstraction (D-08):
  - The runner should support both protocols via the `protocol` discriminator in `PdfUploadSessionDescriptor`, but may implement only one concrete strategy in Phase 26 as long as the code structure makes it trivial to add the second without changing call sites.
  - Do NOT hardcode a vendor; use only `pdfUploadClient` same-origin endpoints.
- Finalize is required (D-10):
  - Runner must call `completePdfUpload` and treat the upload as “done” only if finalize returns success.
- Local-first (D-06):
  - Runner should not block local IDB persistence; it should be safe to run in parallel with existing ingest steps.

Provider readiness + env gating contract (must match Plan 26-01 Task 2 capability contract):
- The runner MUST treat both of these as **local-only** outcomes (not error states):
  - `uploadCapability.configured === false` (env-gated off)
  - `uploadCapability.providerReady === false` (adapter not present / provider not implemented yet)
- In both cases: resolve `{ ok: false, uploadCapability, message? }` and do not emit a “failed” status transition. This is required to satisfy D-01 optional behavior and avoid confusing “upload failed” UI.
  </action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser upload runner → Next API routes | Untrusted client controls timing, retries, and cancel; server must enforce validation and never trust client keys. |
| UI rendering of upload status | Error messages must not leak presigned URLs or sensitive metadata. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|----------|----------|-----------|-------------|-----------------|
| T-26-07 | I (Information disclosure) | Upload progress + errors UI | mitigate | Never render presigned URLs or object keys containing PII; display only coarse uploadId (optional) and bytes/percent. |
| T-26-08 | D (Denial of service) | Auto retry loop | mitigate | Cap retries/backoff and stop retrying on cancel; classify retryable vs non-retryable failures using existing retry helpers. |
| T-26-09 | T (Tampering) | Client-sent session descriptor | mitigate | Treat server as source of truth; client runner never invents keys; server finalize validates allowlist + expiry + size. |
| T-26-10 | R (Repudiation) | User says “upload completed” | mitigate | Require server finalize success before UI marks done; log finalize result with correlation id (uploadId). |
</threat_model>
