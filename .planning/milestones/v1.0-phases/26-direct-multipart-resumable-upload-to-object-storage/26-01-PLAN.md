- Session descriptor includes: `uploadId` (random UUID), `key` (provider object key), `sizeBytes`, `contentType`, `expiresAt` (ISO), and a `protocol` discriminator for multipart vs resumable-session style (per D-08).
- Request/response types for:
  - init (`/api/uploads/pdf/init`) returning the session descriptor + either an immediate part URL or session info, plus a **server-signed `finalizeToken`** (opaque string) that binds `uploadId/key/sizeBytes/contentType/expiresAt` for later verification in `complete`
  - part (`/api/uploads/pdf/part`) returning a presigned URL for a specific partNumber / byteRange
  - complete (`/api/uploads/pdf/complete`) performing server-side finalize/validate and returning a stable `finalized: boolean` + error fields; request MUST include `finalizeToken` from init so server can validate without relying on process memory
  - abort (`/api/uploads/pdf/abort`) best-effort cleanup (no-op ok)

2) Add `src/lib/uploads/pdfUploadContracts.ts` for shared constants/helpers that both client and server use:
- A single source of truth for:
  - allowed content-types (PDF only)
  - key-prefix allowlist (e.g. `uploads/pdf/` — pick a prefix and use it everywhere)
  - random upload id generation and “sanitized suffix” helper (D-03) that:
    - strips path separators and control chars
    - allows only a safe charset for suffix
    - never returns the full original filename unchanged
- A short TTL duration constant and helper to compute `expiresAt` (D-05; exact duration is at Claude’s discretion but must be “hours up to ~1 day”).
 - A helper to create/verify a **server-signed finalize token** using a single server secret (env var) and a stable algorithm (HMAC over a compact JSON payload is fine). The token is strictly for server-side verification of init parameters at complete-time; it is not a bearer auth token.

3) Add `src/lib/uploads/pdfUploadClient.ts` that exposes a minimal API to callers:
- `initPdfUpload({ file, suggestedSuffix })`
- `getPdfUploadPartUrl({ session, partNumber, byteStart, byteEndInclusive })`
- `completePdfUpload({ session, parts, finalizeToken })`
- `abortPdfUpload({ session })`

The client helper must be safe when the feature is not configured OR the provider adapter is not present:

- Define a single, explicit capability contract used across ALL `/api/uploads/pdf/*` routes:
  - Response includes `uploadCapability: { configured: boolean, providerReady: boolean, mode: "local-only" | "direct-upload", reason?: "not_configured" | "provider_not_ready" }`
  - When `configured === false` OR `providerReady === false`, the server MUST respond with a non-error status and `uploadCapability.mode === "local-only"`.
  - The client MUST treat both cases as **local-only** (per D-01) and MUST NOT surface them as an “upload failed” error state.

- Specifically: do NOT use HTTP 501 to represent “provider not implemented” for this phase. The point is env-gated optional behavior (D-01) + vendor-deferred abstraction (D-02), so “adapter absent” is a local-only capability state, not an error.

Do NOT choose a storage vendor in this phase; all network calls are to same-origin routes only.

Implement provider-agnostic Next route handlers that:

- Are **explicitly env-gated** (per D-01) using a single env flag (choose a name like `D2Q_OBJECT_STORAGE_ENABLED`) and return a consistent capability JSON shape when disabled.
- Use `Cache-Control: private, no-store` on JSON responses (match `vision-staging` pattern).

Implement the routes as contracts/stubs (vendor deferred per D-02), but with REAL validations at the boundary (D-12):

`POST /api/uploads/pdf/init`
- Validate JSON body types and return 400 on invalid JSON.
- Validate `contentType` is PDF-only and `sizeBytes` within a reasonable max (use the existing app’s PDF limit unless there’s a separate server limit; keep consistent with local validation).
- Generate `uploadId` and `key = {prefix}/{uploadId}-{sanitizedSuffix}.pdf` (per D-03).
- Return session descriptor including `expiresAt`.
- Return a **server-signed `finalizeToken`** that binds at least: `uploadId`, `key`, `sizeBytes`, `contentType`, `expiresAt`. The client must present this exact token to `complete`.
- Return protocol info sufficient for the client to proceed (even if the part URL generation is stubbed).

`POST /api/uploads/pdf/part`
- Validate session + partNumber / byte range input.
- If upload capability is local-only (either not configured OR provider adapter not present), return a non-error response indicating `uploadCapability.mode === "local-only"` and no URL. Do NOT return 501 for adapter absence in Phase 26.
- The goal is a stable interface shape and env-gated optional behavior, not a working provider.

`POST /api/uploads/pdf/complete`
- This is the **finalize** step (D-10). Even without a vendor, implement validations against the *declared* session:
  - key-prefix allowlist
  - content-type allowlist (PDF)
  - size match against init via **token verification** (server verifies `finalizeToken` signature and compares token payload to request/session)
  - expiry window check via token payload `expiresAt`
- If upload capability is local-only because the provider adapter is not present, treat it the same as “not configured” (D-01): return `finalized: false` with `uploadCapability.mode === "local-only"` and a reason, without surfacing an “upload failed” state.
- If the provider is present but finalize validations fail (D-12), return a clear non-2xx response with a user-displayable message; do not report success.

`POST /api/uploads/pdf/abort`
- Best-effort: accept a session descriptor and return `{ ok: true }` even if provider cleanup is a no-op.

Important: Do **NOT** rely on in-process memory to correlate init→complete. Assume init/part/complete can land on different instances (serverless / multi-instance). Correlation/validation must work purely from the request + server-signed `finalizeToken` (verified server-side with an env secret), so finalize remains correct regardless of instance routing.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|----------|----------|-----------|-------------|-----------------|
| T-26-01 | S (Spoofing) | `/api/uploads/pdf/complete` | mitigate | Require finalize to validate key-prefix allowlist + expiry + declared size/contentType; do not trust client-provided key outside allowlist. |
| T-26-02 | T (Tampering) | Upload key construction | mitigate | Generate key server-side using random `uploadId` and sanitized suffix; never accept raw filenames as keys (D-03). |
| T-26-03 | R (Repudiation) | Upload completion claims | mitigate | Return structured error responses; log server-side finalize failures via existing pipeline log patterns (avoid secrets). |
| T-26-04 | I (Information disclosure) | Keys leaking PII | mitigate | Randomized keys + sanitized suffix; avoid embedding original filename; ensure responses do not echo untrusted filename fields. |
| T-26-05 | D (Denial of service) | Init/part endpoints | mitigate | Enforce max size bounds, short TTL, strict JSON validation, no-store cache-control, and cap request body sizes where possible; avoid any unbounded server-side state. |
| T-26-06 | E (Elevation of privilege) | Presigned URL misuse | mitigate | Enforce key-prefix allowlist at finalize; design contract so presigns are short TTL and scoped to exact key/part (provider-specific later). |
