## Task 2 — Server + client Sentry init files (DSN-gated)

<action>
Add **root** `sentry.server.config.ts` and `sentry.client.config.ts` (same directory as `next.config.ts` unless Sentry docs for your installed major specify `src/` — follow the official manual setup for that major).

Each file calls `Sentry.init({ ... })` with:
- `dsn` from `process.env.SENTRY_DSN` (server) and `process.env.NEXT_PUBLIC_SENTRY_DSN` (client).
- **Disable** SDK when DSN missing/whitespace: `enabled: Boolean(process.env....?.trim())` or equivalent documented pattern.
- **`beforeSend`** on both inits: a function that **returns the event unchanged** for now, with an inline comment exactly: `// Strip PII / secrets in 13-02+; never attach PDF bytes or API keys.`
- Conservative defaults: `tracesSampleRate: 0` unless product explicitly wants performance monitoring in a follow-up (this phase is **errors**, not APM).

Do **not** enable session replay in this phase unless zero config cost — default off.
</action>

<read_first>
- Official Sentry Next.js manual setup for the installed `@sentry/nextjs` major (executor: skim project’s installed package README under `node_modules/@sentry/nextjs` if web unavailable)
</read_first>

<acceptance_criteria>
- `sentry.server.config.ts` imports `@sentry/nextjs` (or `@sentry/node` re-export per docs) and calls `Sentry.init`.
- `sentry.client.config.ts` calls `Sentry.init`.
- `rg "SENTRY_DSN|NEXT_PUBLIC_SENTRY_DSN" sentry.server.config.ts sentry.client.config.ts` matches both filenames.
- `rg "beforeSend" sentry.server.config.ts sentry.client.config.ts` matches both filenames.
