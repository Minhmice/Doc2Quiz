  <action>
Create `src/lib/pdf/layoutBlocksFromTextLayer.ts` exporting:
- **Types**
  - `PdfTextItem` (pageIndex: number, str: string, x: number, y: number, w: number, h: number, fontName?: string, hasEOL?: boolean)
  - `PdfTextLine` (items: PdfTextItem[], text: string, bbox: { x0,y0,x1,y1 })
  - `PdfLayoutBlock` (pageIndex: number, lines: PdfTextLine[], text: string, bbox: { x0,y0,x1,y1 })
  - `BuildLayoutBlocksOptions` with conservative defaults (line merge tolerance, paragraph gap threshold, maxItemsPerPage)

- **Algorithm (deterministic, geometry-driven)**
  1) Normalize items: drop empty/whitespace-only `str`, keep stable ordering key.
  2) Convert pdf.js textContent items into `PdfTextItem` coordinates using pdf.js item transform + viewport (scale=1) so x/y are consistent.
  3) Sort items by \(y\) descending then \(x\) ascending (top-to-bottom, left-to-right); tie-break with original index.
  4) Group into lines by y-proximity (baseline tolerance) and x-proximity to preserve intra-line reading order.
  5) Build block breaks when vertical gap between consecutive lines exceeds a threshold or when an explicit `hasEOL` boundary suggests a paragraph break.
  6) For each line/block, compute bbox and `text` by joining with spaces/newlines (avoid double spaces; preserve short line breaks).

Guardrails:
- Cap processing by `maxItemsPerPage` to avoid pathological PDFs causing runaway CPU/memory.
- Entire module must be pure/deterministic given items + options (no timers, no randomness).
  </action>

    - Large-item pages are safely capped with a clear returned “truncated” signal (either via return metadata or a sentinel block text) without throwing.
  </acceptance_criteria>
