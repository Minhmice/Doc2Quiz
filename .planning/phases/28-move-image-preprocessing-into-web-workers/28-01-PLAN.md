Implement a **single-worker, sequential** resize+JPEG-encode helper (per D-01, D-06) without changing downstream contract (per D-04).

Worker module (`imagePreprocess.worker.ts`):
- Define a small message protocol:
  - Request includes: `requestId`, `bitmap` OR `imageData` (choose one approach; prefer `ImageBitmap` if supported), `maxWidth`, `maxHeight`, `jpegQuality`.
  - Response includes: `requestId`, `ok: true`, `dataUrl` OR `ok: false`, `error: { name, message }`.
- Perform resize logic equivalent to current main-thread behavior (same aspect-preserving scale rules used in `renderPagesToImages.ts` today).
- Encode to JPEG and return `dataUrl` (must start with `data:image/jpeg;base64,`).
- Ensure the worker always releases resources (close `ImageBitmap` when used; allow GC for `ImageData` objects).

Client helper (`encodeJpegInWorker.ts`):
- Export a function like `encodeJpegDataUrlInWorker({ canvas, maxWidth, maxHeight, jpegQuality, signal })` OR a more general "input is canvas/ImageBitmap" shape (Claude discretion), but:
  - MUST accept an `AbortSignal` and reject with an `AbortError` quickly when aborted (per D-05).
  - MUST manage **one worker instance** and queue requests sequentially (per D-06).
  - MUST expose a capability check (e.g. `canUseImagePreprocessWorker(): boolean`) that returns false when `Worker` or required image APIs are missing (per D-03).
  - MUST fail-safe: if worker construction or a request fails, the caller can fall back to main-thread encode.
- Add a **small internal timeout guard** only if needed to avoid hanging promises (keep it conservative; do not change UX; log to `pipelineLog` only if available).
