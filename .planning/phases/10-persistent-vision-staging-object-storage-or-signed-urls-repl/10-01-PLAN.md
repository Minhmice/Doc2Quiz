## Trust boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Next.js route handlers | Untrusted `dataUrl` payload crosses into server decode + storage |
| Server → Vercel Blob API | RW token used server-side only; blob URLs returned to client/upstream |
| Upstream LLM vendor → HTTPS URL | Model servers fetch staged image bytes from public blob or same-origin GET |

## STRIDE register (ASVS L1 style)

| Threat ID | Category | Component | Disposition | Mitigation plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01-1 | D | POST `/api/ai/vision-staging` | mitigate | Cap decoded bytes at `VISION_STAGING_MAX_BYTES`; reject non-data URLs and invalid base64 (existing). Document lack of rate limiting in README; optional future: middleware rate limit / auth phase. |
| T-10-01-2 | D | Memory store | mitigate | TTL + max entry eviction reduces memory exhaustion (existing `TTL_MS`, `MAX_ENTRIES`). |
| T-10-01-3 | I | Staging responses | mitigate | `Cache-Control: private, no-store` on GET (existing); Task 1 adds same for POST JSON so caches do not retain staging metadata. |
| T-10-01-4 | I | Error logs | mitigate | Keep `console.error` messages generic; avoid logging full request body or token; log error **message** or code only if tightening logs. |
| T-10-01-5 | E | Public blob URL | accept + document | UUID v4 + random `id` reduces guessability; URL is intentionally fetchable by vendor—residual risk anyone with URL reads image until blob deleted manually or by policy. README states public + no app TTL. |
| T-10-01-6 | T | `BLOB_READ_WRITE_TOKEN` | mitigate | Server env only; never expose to client; never commit to repo (README points to dashboard). |
| T-10-01-7 | E | Path traversal / key injection | mitigate | `visionStagingBlobPathname` uses fixed prefix + server-generated UUID only; GET rejects ids failing `isVisionStagingId` (existing). |
</threat_model>
